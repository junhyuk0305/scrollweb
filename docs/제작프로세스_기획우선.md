# 제작 프로세스 — 기획 우선 (Content-first, 3단계 분리)

> **왜 이 문서**: "효과부터" 만들면 애니메이션 갤러리가 된다. 기업 홈페이지는 *고객을 행동으로 이동시키는 논증*이고,
> 애니메이션은 그 논증을 **증폭하는 종속 수단**이다. 그래서 단계를 물리적으로 분리한다:
> **① 기획(콘텐츠) → ② 개발 명세(콘텐츠×모션 매핑) → ③ 구현/QA → ④ 배포·인계(GitHub Pages).** 앞 단계 산출물 없이는 다음 단계 진입 금지.
>
> 연결: 발굴 엔진 = [기획프롬프트_전문가발굴.md](기획프롬프트_전문가발굴.md) · 명세 양식 = [../_catalog/PLAN.template.md](../_catalog/PLAN.template.md)
> · 파일구조/반응형 = [제작표준_파일구조와기획.md](제작표준_파일구조와기획.md) · tier 효과 하한 = [../내부_tier퀄리티기준.md](../내부_tier퀄리티기준.md)

---

## 대원칙 (벽에 붙일 것)

1. **콘텐츠가 먼저, 모션은 나중.** 모션은 *없어도 메시지가 전달되는* 위에 얹는 증폭이다.
2. **모든 모션은 정당화된다.** "이 효과가 *어떤 콘텐츠 목표*(이해/감정/위계/주목)를 돕는가?" 답 못 하면 뺀다. 장식 금지.
3. **단계 게이트.** Phase N은 Phase N-1의 산출물(아티팩트)을 입력으로만 시작한다.
4. **모션 없이도 완성.** reduced-motion·CDN실패 상태가 곧 "콘텐츠 완성본". 그 위에 모션은 진행적 향상(progressive enhancement).

## Phase 1 — 기획 (콘텐츠 브리프)

- **하는 일**: [기획프롬프트_전문가발굴.md](기획프롬프트_전문가발굴.md) 실행 → 비즈니스/고객/이의/증거/메시지/IA/전환을 발굴.
- **보조 스킬**: `office-hours`(스코프), `content-strategy`+`copywriting`(카피), 업종 리서치([저단가-웹외주-통합보고서.md](저단가-웹외주-통합보고서.md)).
- **산출물(아티팩트)**: `projects/<slug>/BRIEF.md` (= 콘텐츠 브리프).
- **이 단계 금지어**: 색·폰트·레이아웃·스크롤·애니메이션·GSAP. (한 번이라도 나오면 반려)
- **게이트 ✅ (브리프 통과 기준)**: 발굴 프롬프트 §4 체크리스트 전부 — 전환행동 1개, 장식 섹션 0, 카피 실제수준, 이의 전부 섹션에서 해소, 규제 금지표현 0.

> 게이트 통과 전에는 단 한 줄의 CSS/효과도 정하지 않는다.

## Phase 2 — 개발 명세 (콘텐츠 × 모션 매핑)

브리프를 "지을 수 있는 명세"로 번역한다. **여기서 처음으로 디자인·모션이 등장**하되, 항상 콘텐츠에 매여서.

1. **IA 확정 → 파일/페이지 구조**: 30=원페이지 / 50=원페이지~3p+메뉴 nav / 80=≤5p 멀티+풀 nav. (제작표준 §1 폴더 규칙)
2. **섹션 명세**: 브리프 §6 논증표의 각 행에 대해 →
   - 레이아웃 의도(좌우/풀블리드/그리드 — *왜 이 배치가 메시지를 돕는지*)
   - 컴포넌트(카드/슬라이드/표/폼/지도)
   - 콘텐츠 슬롯(확정 카피·이미지 자리·증거)
3. **모션 매핑 표 (★ 이 단계의 핵심)** — 섹션마다:

   | 섹션 | 콘텐츠 목표 | 그 목표를 돕는 모션 | data-fx | 폴백(모션 OFF 시) |
   |---|---|---|---|---|
   | 히어로 | "여기다"를 3초에 각인 | 텍스트가 콘텐츠 위계 순으로 등장 | fade-up(순차) | 즉시 표시 |
   | 실적 | 신뢰를 수치로 | 숫자가 차오르며 주목 | count-up | 최종 숫자 표시 |
   | 시공과정 | 과정을 '여정'으로 이해 | 가로 스크롤로 단계 진행 | hscroll(pin) | 세로 스택 |

   - **밀도 기준선(전 tier)**: **모든 요소는 최소 진입 리빌**(읽기 순서 주목 = 정당화됨) + 모든 이미지 패럴랙스 + 마퀴/카운트업 등 보조 모션. "밋밋한 정적 섹션 0." (레퍼런스: clinic-v2·demo-7·camping — [../내부_tier퀄리티기준.md](../내부_tier퀄리티기준.md) §0)
   - **주인공(시그니처) 효과**: pin/시퀀스/가로스크롤 등 무거운 효과는 *콘텐츠 목표*에 매여야 하고 tier 한도 내(30:1~2 / 50:2~3 / 80:3, 성능예산 pin≤3).
   - 효과는 [../_catalog/00_scroll.md](../_catalog/00_scroll.md) 55종에서 골라 엔진 훅으로 호출. tier 하한은 [../내부_tier퀄리티기준.md](../내부_tier퀄리티기준.md) §1·§2.
4. **디자인 토큰·팔레트 선택**: `_catalog`에서 스타일1·레이아웃1·UX 조합(제작표준 §2 / PLAN.template §5·§6).
5. **폼·지도·규제·SEO·교체목록**: PLAN.template §7~§9.
- **산출물**: `projects/<slug>/PLAN.md` (PLAN.template 기반, **모션 매핑 표 포함**).
- **게이트 ✅**: 모든 모션이 콘텐츠 목표에 1:1로 매여 있음 · 주인공 효과 ≤2 · 전 효과 reduced-motion 폴백 정의 · tier 하한 충족.

## Phase 3 — 구현 & QA

- **하는 일**: `_engine`(또는 프로젝트 공통 css/js)로 PLAN.md 그대로 구현. 콘텐츠(HTML) → 폴백 확인 → 그 위에 모션 적용.
- **보조 스킬**: `page`(스캐폴드), `frontend-design`(룩), `webapp-testing`+`qa`(반응형·버그), `review`/`ship`.
- **순서 강제**: ① 모션 0 상태로 콘텐츠/반응형 완성 → ② 폴백 검증 → ③ 모션 레이어 추가 → ④ 모션 OFF로 재검증.
- **게이트 ✅ (구현 완료)**: 제작표준 §3 반응형 계약 + 내부 tier기준 §4 셀프체크 전부. Lighthouse 90+(30은 85+). → 실제 납품·배포는 Phase 4.

## Phase 4 — 배포 & 인계 (GitHub Pages)

QA 통과본을 무료 호스팅으로 띄우고 **도메인·HTTPS를 연결한 뒤 고객에게 소유권째 넘긴다.** 이 단계의 산출물은 "라이브 URL + 인계 패키지".

- **왜 GitHub Pages가 맞나**: 우리 스택은 정적(HTML/CSS/JS) + 폼은 구글시트/web3forms로 외부화 → GitHub Pages의 "정적만 가능(서버·DB 불가)" 제약과 **정확히 일치**. 무료 + 무료 HTTPS + 상업적 사용 허용.

### 소유권 원칙 (★ 분쟁 예방 — 먼저 못박기)
- **도메인 = 고객 명의로 구매·결제** (가비아·후이즈·Cloudflare 등, .com 기준 연 1~2만원). 우리 책임 아님.
- **저장소 = 고객 GitHub 계정/조직 소유.** 우리는 collaborator로 들어가 배포만 → 우리 계정에 묶이면 이전 분쟁. "고객 계정 무료 호스팅" 노선의 핵심.

### 배포 절차
1. 고객 계정에 **public 저장소** 생성 → `projects/<slug>/` 산출물 push
2. **Settings → Pages → Branch: `main` / root** 게시 → 임시 URL `<계정>.github.io/<repo>` 확인
3. 도메인 구매처 **DNS 레코드** 입력:

   | 연결 형태 | 레코드 |
   |---|---|
   | 루트 도메인 `example.com` | `A` 4개 → `185.199.108.153` · `.109.153` · `.110.153` · `.111.153` |
   | `www` 서브도메인 | `CNAME` → `<계정>.github.io` |

   > IP 4개는 배포 시점에 GitHub Pages 공식 문서로 1회 재확인(가끔 갱신). 한국 도메인(.co.kr)·가비아·후이즈 모두 A/CNAME 지원.
4. **Settings → Pages → Custom domain** 입력(저장소에 `CNAME` 파일 자동 생성) → **Enforce HTTPS** 체크 → Let's Encrypt 인증서 자동 발급
5. DNS 전파(수십 분~수 시간) 후 `https://` 정상 접속 확인

### 인계 패키지 (고객에게 함께 주는 것)
- **주석 처리된 소스**(저장소 그대로) + **README**: 텍스트·이미지·연락처 교체 방법
- **라이브 URL** + HTTPS 적용 스크린샷
- **교체 가이드**: 이미지·연락처·사업자번호 위치 = PLAN §9 교체목록과 1:1
- **수정·유지보수 정책**: 납품 후 30일(또는 N회) 수정, 이후 월 유지보수 옵션

- **게이트 ✅ (인계 완료 = 진짜 납품)**:
  - 커스텀 도메인 `https` 정상 + Enforce HTTPS ON
  - 6개 뷰포트 **라이브에서 재확인**(제작표준 §3)
  - 저장소·도메인 **소유권 고객 측 확인**
  - 교체목록 전부 반영 + README 인계 완료

### 제약 & 대안
- 정적만 가능 / 저장소 1GB·대역폭 100GB월 소프트리밋(소상공인 충분) / 비공개 소스는 private+Pages가 유료 → 보통 public.
- **호스팅 = GitHub Pages 확정** ([개발계획](개발계획_데모포트폴리오.md)). 폼은 구글시트(Apps Script) 또는 이메일. 차순위 대안은 Cloudflare Pages(도메인까지 한 곳에서 사면 연결이 더 매끄러움, 동일 조건 충족).

## 기존 데모에 적용 (리팩터링 규칙)

지금까지의 demo-1~8/clinic/compare-stay는 "Phase 3만 있고 1·2가 없는" 상태 → **역설계로 보강**:
1. 각 데모에 대해 **BRIEF.md를 사후 작성**(이 사이트가 무슨 비즈니스를 어떻게 설득하는지).
2. 현재 섹션을 논증표에 대입 → 목적 없는 섹션 제거/통합, 빠진 설득 단계(이의 해소·증거) 추가.
3. 모션 매핑 표로 **장식 모션 정리**, 부족 tier는 하한까지 보강.
4. Phase 3 순서로 재구현 → QA.

> 신규 데모도 동일하게 **BRIEF.md → PLAN.md → 구현** 순서로만 만든다. (효과부터 만드는 일 금지)
