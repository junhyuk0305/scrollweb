# 제작 표준 — 통일 파일구조 · 페이지 기획 · 반응형 필수

> 데모를 몇 개 만들어보니 드러난 3가지 표준화 과제를 확정한다.
> ① **상품별 통일 파일구조** ② **페이지 기획 md(기획서)** ③ **반응형 = 절대 필수**.
> 기준: [상품구성_스크롤홈페이지.md](상품구성_스크롤홈페이지.md) · [개발계획_데모포트폴리오.md](개발계획_데모포트폴리오.md). 스택: plain HTML/CSS/JS + GSAP(CDN) + 무료 호스팅 + 폼은 구글시트/이메일.

## 1. ① 통일 파일구조 — "엔진은 통일, 룩은 매번 다르게" (anti-template)

> ⚠️ **핵심 원칙**: 공통화가 "하나의 템플릿"이 되면 모든 사이트가 똑같아 보인다 = 실패.
> 그래서 **통일하는 것(기능·폴더)**과 **매번 달라지는 것(룩·구성)**을 물리적으로 분리한다.
> 우리에겐 **애니 55종(코어30+시그니처16+확장9) · 디자인스타일30 · 레이아웃15 · UX15**가 있다. 이 팔레트를 업종·기획에 맞게 **골라 조합**하는 게 다양성의 원천이고, 엔진은 그 조합을 "실행"만 한다.

### 3계층 모델

| 계층 | 통일? | 무엇 | 어디 |
|---|---|---|---|
| **A. 엔진(Engine)** | ✅ 통일·불변 | 기능만. GSAP/Lenis 부트, ScrollTrigger 동기화, a11y/모바일 폴백 정책, 반응형 프리미티브, 폼. **룩 0** | `_engine/` |
| **B. 팔레트(Catalog)** | ✅ 공유 메뉴 | 효과 55 + 스타일30 + 레이아웃15 + UX15. **선택지**(코드/레퍼런스) | `_engine/fx/` + `_catalog/` |
| **C. 스킨(Skin/구성)** | ❌ 매번 다름 | 토큰(색·폰트·간격)·레이아웃 선택·섹션 구성·카피. **여기서 룩이 결정** | `projects/<slug>/` |

→ 두 사이트가 엔진 코드를 100% 공유해도, **고르는 효과 + 토큰 + 레이아웃이 다르면 전혀 달라 보인다.** 엔진은 보이지 않는 배관.

```
06_websell/
├─ CLAUDE.md                    ← AI 운영 지침 (루트엔 이것만)
├─ docs/                        ← 기획·전략 문서 (21.md, 상품구성, 개발계획, 본 표준, 저단가보고서, 프로젝트개요)
├─ portfolio/                   ← 포트폴리오 메인 사이트 (판매 쇼케이스 허브)
│  ├─ index.html  css/  js/     ← 인터랙티브 필름 스트립(전체 데모 카드)
│  └─ works/                    ← 데모·작업물 전부 (demo-1~8, clinic-v1~5, yunseul-stay, labor-service)
├─ _engine/                     ← A. 통일·불변 (기능만, 룩 0)  ※Phase 0에서 생성
│  ├─ core.js                   ← GSAP/Lenis 부트 + ST 동기화 + 앵커 스크롤 + a11y/모바일 폴백 정책
│  ├─ fx/                       ← B. 효과 레지스트리 (data-fx="…" 로 호출, ~55종)
│  │  ├─ hero.js type.js image.js layout.js bg.js data.js signature.js
│  ├─ form.js                   ← 구글시트(Apps Script) / 이메일(web3forms)
│  └─ base.css                  ← 리셋 + 반응형 프리미티브 + 토큰 "슬롯"(기본값 중립, 강제 오버라이드)
│
├─ _catalog/                    ← B. 선택 메뉴 (기획 시 고름. 코드 아님)
│  ├─ 00_scroll.md              ← 애니 55종 ↔ fx-id 매핑표
│  ├─ 1_디자인스타일-30종.md
│  ├─ 2_레이아웃구조-15종.md
│  ├─ 3_UX인터랙션-15종.md
│  ├─ 00_scroll.html            ← 효과 라이브 데모
│  ├─ PLAN.template.md  README.template.md
│
└─ projects/<slug>/             ← C. 매번 다름 (스킨·구성)  예: noon-fashion, nuve-hotel
   ├─ index.html                ← _engine 참조 + data-fx 지정. 섹션 구성은 자유
   ├─ about.html / rooms.html…  ← STANDARD·PRO 멀티페이지
   ├─ assets/
   │  ├─ css/style.css          ← ★이 프로젝트 토큰·레이아웃 = 룩 결정 지점
   │  ├─ js/app.js              ← 어떤 fx를 어디에 걸지 등록/조합
   │  └─ img/                   ← 고객 이미지(교체 대상)
   ├─ PLAN.md  README.md
```

**규칙 (anti-template 보장)**
1. **엔진은 룩을 모른다.** `_engine`엔 색·폰트·섹션 레이아웃이 **없다**. `data-fx="shrink-lock"`처럼 **모션만** 건다. 외형은 100% 프로젝트 CSS가 결정 → 같은 효과도 스킨마다 다르게 보임.
2. **마스터 template.html 금지.** "복붙하는 한 장"을 두지 않는다. 대신 (선택) 구조용 스니펫은 **색·레이아웃 의견이 없는 뼈대**만 제공하고, 프로젝트 토큰으로 매번 새로 입힌다.
3. **base.css 토큰은 슬롯.** 기본 팔레트를 사실상 안 준다(중립). 프로젝트 `style.css`가 `:root` 토큰을 **반드시 새로 정의**해야 함 = 디폴트 룩 답습 방지.
4. **다양성은 PLAN에서 설계.** 업종·기획에 맞춰 `_catalog`에서 **스타일1 + 레이아웃1 + UX 몇 + 효과 N**을 골라 조합(PLAN.md §6). 엔진은 그 선택을 실행만.
5. **통일되는 것은 딱 둘**: ⓐ 폴더/파일 위치 ⓑ 엔진 동작·a11y·반응형 정책. 그 외(룩·구성·카피·효과 선택)는 전부 자유.
6. **tier 형태**: BASIC=`index.html` 1장 / STD·PRO=멀티페이지. 폴더 구조는 tier 무관 동일.
7. **데모도 이관**: demo-1~8 → `projects/`로(레퍼런스). 기존 인라인은 STD↑에서 분리.

> 요약: **통일 = 배관(엔진)·서랍(폴더). 개성 = 팔레트 조합(스킨).** 템플릿이 아니라 "악기(엔진) + 악보집(팔레트)"이고, 연주(프로젝트)는 매번 다르다.

## 2. ② 페이지 기획 md (PLAN.md) — 표준 템플릿

코드 전에 **반드시** `projects/<slug>/PLAN.md`를 먼저 채운다(spec-driven). 실파일: [_catalog/PLAN.template.md](../_catalog/PLAN.template.md). 구성:

1. **프로젝트 메타** — 업종 · tier · 브랜드 · 톤&무드 · 레퍼런스 URL · 마감일
2. **타겟 & 목표** — 방문자, **핵심 전환 행동 1개**(문의/예약/전화), 성공기준
3. **정보구조(IA)** — 페이지·섹션 목록 + **총 섹션 수(= 견적 기준)**
4. **섹션별 기획표** — `섹션명 | 목적 | 콘텐츠 | data-fx 효과(55종) | CTA`
5. **디자인 토큰** — 핵심색 3~4 · 폰트 2 · 라운드/간격 무드
6. **팔레트 선택(다양성 설계)** — `_catalog`에서 **디자인스타일1 + 레이아웃1 + UX 몇 + 효과 N** 조합. tier 한도(BASIC 기본세트 / STD 시그니처1 / PRO 맞춤2)
7. **폼/연동** — 구글시트 or 이메일, 수집 필드
8. **SEO/메타 + 규제** — title/desc/OG/파비콘 + (의료·법무·학원) 금지표현 체크
9. **교체 목록** — 판매 시 교체할 이미지·연락처·사업자번호 리스트
10. **반응형 & 납품 체크리스트** — §3 그대로

> office-hours로 1·2번을, copywriting으로 4번 콘텐츠를, seo-audit로 8번을 채우면 빈칸이 빨리 닫힌다.

## 3. ③ 반응형 = 절대 필수 (RESPONSIVE CONTRACT)

> **모든 산출물은 아래를 100% 통과하지 못하면 납품 불가.** 데모·포트폴리오 포함 예외 없음.

**필수 뷰포트(깨짐 0)**: `320 · 375 · 414 · 768 · 1024 · 1440`px
**하드 규칙**
- 가로 스크롤(`overflow-x`) **절대 금지** — 모든 페이지 `body{overflow-x:hidden}` + 원인 요소 제거
- 모바일 우선. 폰트 `clamp()`, 이미지 `max-width:100%;display:block`
- 터치 타겟 **≥44px**, 고정 CTA(스티키/플로팅) 모바일 노출
- 데스크톱 nav → 모바일 햄버거/스티키 CTA 전환
- 가로 스크롤·핀 섹션은 모바일에서 **세로 폴백** 또는 비활성
- `prefers-reduced-motion` 폴백 필수(애니 꺼도 콘텐츠 정상)
- 표/그리드는 모바일에서 1열 스택

**검증(자동)**: `webapp-testing`(Playwright)로 6개 뷰포트 스크린샷 → 깨짐 확인, `qa` 스킬로 버그 자동검출+수정. PLAN.md 체크리스트에 결과 기록.

**납품 체크리스트(README.md에 복사)**
- [ ] 6개 뷰포트 깨짐 0 (스크린샷 첨부)
- [ ] 가로 스크롤 없음
- [ ] Lighthouse(모바일) 성능·접근성 90+
- [ ] reduced-motion 폴백 동작
- [ ] 폼 → 구글시트/이메일 수신 확인(STD↑)
- [ ] 교체 목록 모두 반영(이미지·연락처·사업자번호)
- [ ] 무료 호스팅 배포 + 도메인 연결 + SSL

## 4. 제안 실행 순서

1. **이 표준 확정** → `_catalog/PLAN.template.md`·`README.template.md` 생성 (완료)
2. **`_engine/` 구현 = 개발계획 Phase 0** — `core.js`(부트·동기화·a11y/모바일 정책) + `base.css`(리셋·반응형 프리미티브·토큰 슬롯) + `fx/`(효과 레지스트리, `data-fx` 호출) + `form.js`. **룩 0, 기능만.**
3. `_catalog/00_scroll.md`의 55종에 **fx-id 매핑** 부여(예: `shrink-lock`, `kinetic-marquee` …) → `fx/`에 점진 구현
4. 신규 데모/고객은 처음부터 `projects/<slug>/` 구조로(엔진 참조 + 토큰·레이아웃·효과 선택)
5. 기존 demo-1~8 → `projects/`로 이관하며 반응형 컨트랙트 재검증(qa)
6. 클라이언트 작업: `office-hours → PLAN.md → copywriting → page 스캐폴드 → frontend-design → qa(반응형) → ship`
